# エリートエージェント・ドキュメンテーション・フレームワーク：グローバルガイドライン (PO版)

## 第1章：理念とコアバリュー

**エリートエージェント・ドキュメンテーション・フレームワーク**（Elite Agent Documentation Framework）は、5つの専門エージェントが連携し、生のアイデアを完全なMVPプロジェクト資料およびプロフェッショナルなUI/UXデザインドキュメントへと変換するクローズドなプロセスです。

### 1.1 JTBD（片付けるべき用事）の理念
すべての製品は、単なる「機能」として存在するのではなく、ユーザーの核となる「仕事（ジョブ）」を解決するために作られます。当フレームワークでは、エージェントに対し常に「この製品は何をするために雇われたのか？」と問い続け、その目的を果たさない不要な要素をすべて排除することを求めています。

### 1.2 5つの専門エージェントモデル
1. **LearnAgent (リサーチ & 戦略)**: 競合他社や市場のウェブサイトから知識を収集します (KWSR構造)。
2. **Agent 1 (インテイク & 戦略)**: `What & Why`（何のために、なぜ）を定義します。BRD/PPD/SDKを作成。
3. **Agent 2 (アーキテクチャ)**: `How`（どのように）を定義します。技術的な構成（SDD）を設計。
4. **Agent 3 (詳細設計 & QA)**: `Quality & Detail`（品質と詳細）を定義します。SRS/PRD/テストケースを作成。
5. **Agent 4 (UI/UXデザイナー)**: `Look & Feel`（見た目と使い心地）を定義します。デザインスペック（Design Specs）を作成。

## 第2章：製品開発ロードマップ

### フェーズ 0: 知識獲得 (LearnAgent)
競合サイトをリバースエンジニアリングし、実際のユーザーフロー、ビジネスモデル、機能、および非機能要件（NFR）を取得します。

### フェーズ 1: ビジネス戦略 (Agent 1)
「アイデア主導型（Idea Driven）」、「提案主導型（Proposal Driven）」、または「ソース主導型（Source Driven、LearnAgentからの情報）」の3つのシナリオを通じて、問題、ユーザー、JTBDを明確にします。
> [!IMPORTANT]
> **BRD（ビジネス要件定義書）には以下の内容が必須です**: 詳細なビジネスプロセスフロー、および非機能要件（NFR）の抽出。

### フェーズ 2: 技術設計 (Agent 2)
Agent 1 のBRDに基づき、アーキテクチャ、データベーススキーマ（DB Schema）、およびAPI設計を行います。

### フェーズ 3: ドキュメント化 & QA (Agent 3)
詳細な要件（SRS/PRD）をシステム化し、業務ロジックと同期したテストケースを作成します。

### フェーズ 4: ビジュアルデザイン (Agent 4)
要件をインターフェースに変換します。
> [!IMPORTANT]
> **デザインスペック（Design Specs）には以下の内容が必須です**: 番号付けされた画面一覧（Screen Inventory）を作成し、PRD/BRD内のユーザーフローと直接紐付けます。

## 第3章：標準出力構造

各プロジェクトの `Output/[Project_Name]/` 内には、以下のディレクトリを構成しなければなりません：

- `LearnAgent/`: 生のリサーチデータと競合分析。
- `Agent 1/`: `[Project]_BRD.md`, `[Project]_PPD.md`, `[Project]_SDK.md`。
- `Agent 2/`: `[Project]_Solution_Design.md`。
- `Agent 3/`: `MVP_PRD.md`, `[Project]_SRS.md`, `MVP_Test_Cases.md`。
- `Agent 4/`: `Design_Specs.md`, `Screens/`。

プロジェクトのルートディレクトリ：
- `Master_Project_Index.md`: すべての資料へのナビゲーションマップ。

## 第4章：ドキュメントの役割と具体的要件

### 4.1 Agent 1: 3つの重要ドキュメント
- **BRD**: 業務の羅針盤。ビジネスプロセスフローと非機能要件（NFR）の記述が**必須**です。
- **PPD**: 市場での位置付けとターゲットユーザーのペルソナ。
- **SDK**: ソーシャルプルーフ（社会的証明）と販売用コンテンツのセット。

### 4.2 Agent 4: エリート標準デザインスペック
- **画面一覧 (Screen Inventory)** が必須：設計が必要なすべての画面のリスト。
- **ユーザーフロー (User Flow)** との整合：ユーザー体験のジャーニーにおいて一歩も欠けていないことを保証。

## 第5章：管理と運用

1. **ドキュメント更新ポリシー (Living Document Policy)**: ルールが変更された場合、このガイドラインは即座に更新されなければなりません。
2. **厳格な引き継ぎ (Strict Handover)**: 後続のエージェントは前任のエージェントのデータを継承します。スコープの大きな変更はPO（プロダクトオーナー）の承認が必要です。
3. **一貫性ルール (Consistency Rule)**: プロジェクト名、ファイル名、用語は、5つのエージェント全体で一貫していなければなりません。
4. **相対パスルール (Relative Path Rule)**: 各ドキュメント内および `Master_Project_Index.md` 内のすべてのリンクには、**相対パス（Relative Path）** を使用しなければなりません。ポータビリティ（持ち運びやすさ）を確保するため、絶対パスや `file:///d:/` 形式は絶対に使用しないでください。
