# Elite Agent Documentation Framework: グローバルガイドライン (PO 版)

## 第1章：哲学とコアバリュー

**Elite Agent Documentation Framework** は、5つの専門エージェントで構成される一連のプロセスです。これらが連携することで、生のアイデアを完全な MVP プロジェクト資料および専門的な UI/UX デザインドキュメントへと変換します。

### 1.1 JTBD (Job to Be Done) の哲学
すべての製品は、単なる「機能」のために存在するのではなく、ユーザーの核となる「用事 (Job)」を解決するために作成されます。このフレームワークでは、すべてのエージェントが「この製品は何を解決するために採用されたのか？」と常に自問し、その目標に貢献しない不必要な要素を排除することを義務付けています。

### 1.2 5つの専門エージェントモデル
1. **LearnAgent (調査・戦略)**: 競合他社や市場のウェブサイトから知識を収集します。
2. **Agent 1 (インテーク・戦略)**: 「何を、なぜ (What & Why)」を定義し、BRD/PPD/SDK を構築します。
3. **Agent 2 (設計)**: 「どのように (How)」を定義し、技術構造 (SDD) を構築します。
4. **Agent 3 (詳細化)**: 「品質と詳細 (Quality & Detail)」を定義し、PRD/SRS/テストケースを構築します。
5. **Agent 4 (UI/UX デザイナー)**: 「見た目と操作感 (Look & Feel)」を定義し、テザインスペックを構築します。

## 第2章：製品開発ロードマップ

### フェーズ 0: 知識獲得 (LearnAgent)
競合サイトをリバースエンジニアリングし、実際のユーザーフロー、ビジネスモデル、機能、および非機能要件 (NFR) を抽出します。

### フェーズ 1: ビジネス戦略 (Agent 1)
「アイデア主導 (Idea Driven)」、「提案主導 (Proposal Driven)」、または「ソース主導 (Source Driven/LearnAgent由来)」の3つのシナリオを通じて、問題、ユーザー、JTBD を明確にします。
> [!IMPORTANT]
> **BRD（業務要件定義書）の必須事項**: 詳細なビジネスプロセスフローおよび非機能要件 (NFR) の定義。

### フェーズ 2: 技術設計 (Agent 2)
Agent 1 の BRD に基づいて、アーキテクチャ、DB スキーマ、および API デザインを設計します。

### フェーズ 3: ドキュメント作成と品質保証 (Agent 3)
詳細要件 (SRS/PRD) を体系化し、業務ロジックと同期したテストケースを作成します。

### フェーズ 4: ビジュアルデザイン (Agent 4)
要件をインターフェースに変換します。
> [!IMPORTANT]
> **デザインスペックの必須事項**: 画面インベントリ (Screen Inventory) を作成し、PRD/BRD 内のユーザーフローと直接紐付ける必要があります。

## 第3章：標準出力構造

各プロジェクトは `Output/[Project_Name]/` 内で以下のディレクトリに整理される必要があります：

- `LearnAgent/`: 生の調査データと競合分析。
- `Agent 1/`: `[Project]_BRD.md`, `[Project]_PPD.md`, `[Project]_SDK.md`。
- `Agent 2/`: `[Project]_Solution_Design.md`。
- `Agent 3/`: `MVP_PRD.md`, `[Project]_SRS.md`, `MVP_Test_Cases.md`。
- `Agent 4/`: `Design_Specs.md`, `Screens/`。

プロジェクトのルートディレクトリ：
- `Master_Project_Index.md`: 資料全体のナビゲーションマップ。

## 第4章：ドキュメントの役割と具体的な要件

### 4.1 Agent 1: 3つの強力なドキュメント
- **BRD**: 業務の羅針盤。ビジネスプロセスフローと非機能要件 (NFR) の記述が**必須**です。
- **PPD**: 市場ポジショニングとターゲットユーザー像。
- **SDK**: ソーシャルプルーフ（社会的証明）の素材とセールスコンテンツ。

### 4.2 Agent 4: デザインスペック (Standard Elite)
- **画面インベントリ (Screen Inventory)**: 設計が必要なすべての画面リストが必須です。
- **ユーザーフローとの一致**: ユーザー体験の流れの中でステップが漏れていないことを確認してください。

## 第5章：管理と運用

1. **Living Document Policy**: このガイドラインは、規則が変更された際に即座に更新される必要があります。
2. **Strict Handover (厳格な引き継ぎ)**: 後続のエージェントは前段のエージェントのデータを継承します。大幅なスコープ変更には PO の承認が必要です。
3. **Consistency Rule (一貫性の規則)**: プロジェクト名、ファイル名、用語は5つのエージェント全体で一貫させる必要があります。
4. **Relative Path Rule (相対パスの規則)**: すべてのリンクおよび `Master_Project_Index.md` では、必ず**相対パス**を使用してください。絶対パスや `file:///d:/` 形式は、ドキュメントのポータビリティを確保するために絶対に使用しないでください。
