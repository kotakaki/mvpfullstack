# Rules: Signal Log Template (シグナルログ・テンプレート)

`Signal_Log.md` は、シグナルをSNS（LinkedIn/X）やブログのコンテンツに変換することを目的とする場合、以下の構造を使用するべきです。

```md
# Signal_Log - <テーマ名>

## 背景 (Context)
- テーマ:
- 範囲:
- 期間:
- コンテンツの目標:
- 優先読者層:

## Signal 01
- 日付:
- テーマ:
- 情報源名:
- ソースタイプ:
- URL:
- シグナル / タイトル:
- 要約:
- 注目すべき理由:
- 初期信頼性:
- タグ:
- 適した読者層:
- 適したコンテンツチャネル:
- 潜在的なSNSアングル:
- 潜在的なブログアングル:
- 使用可能な事実 / 引用:
- 注目すべき言語 / フレーズ:
- コンテンツ化の準備状態:

## Signal 02
... (同上の繰り返し)

## 初期シグナルのクラスター（グループ）

### クラスター A - <クラスター名>
- 該当シグナル:
- 観測結果:
- コンテンツへの含意:
- 優先順位:
```

## 運用規則
- `日付`、`情報源名`、`URL`、および `初期信頼性` を省略してはいけません。
- `要約` にはソースから直接得られた事実のみを記載し、早期に個人的な推奨事項（レコメンデーション）を混ぜないでください。
- `注目すべき理由` には、そのシグナルがプロダクト、マーケティング、またはコンテンツ作成においてなぜ重要なのかを明記してください。
- `適した読者層` は具体的に指定します。例：
  - `B2Bファウンダー`
  - `プロダクト・マーケター`
  - `コンテンツ・リード`
  - `SEOマネージャー`
- `適したコンテンツチャネル` は簡潔に選択します：
  - `LinkedIn投稿`
  - `X(Twitter)スレッド`
  - `ウェブサイト・ブログ`
  - `ニュースレター`
  - `その他（併用）`
- `潜在的なSNSアングル` は、記事全体ではなく、1文程度の要点として記述してください。
- `潜在的なブログアングル` は、以下のような明確な構成指針を示してください：
  - `解説記事 (Explainer)`
  - `トレンド分析 (Trend Analysis)`
  - `意見表明 (Opinionated take)`
  - `ハウツー (How-to)`
  - `ケーススタディ (Case-based article)`
- `使用可能な事実 / 引用` には、ソースから直接得た数値、事実、または強力な引用句のみを記載し、言葉を捏造してはいけません。
- `注目すべき言語 / フレーズ` は、市場や読者が使っているリアルな表現を保持するために使用します。
- `コンテンツ化の準備状態` は3段階で評価します：
  - `すぐに作成可能`
  - `さらなる検証が必要`
  - `継続監視のみ`
- クラスター化が可能で、コンテンツシリーズの方向性が見えている場合は、積極的に `初期シグナルのクラスター` セクションを追加してください。
