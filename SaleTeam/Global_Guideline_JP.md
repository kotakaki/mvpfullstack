# SaleTeam: 総合ガイドライン

## 1. 目標
`SaleTeam` は、以下の活動に役立つデータの収集、標準化、および拡充（Enrichment）を専門としています：
- リード（見込み客）の生成
- アカウント（企業）調査
- セールス活動のパーソナライゼーション（個別最適化）
- 顧客の育成（ナーチャリング）

目標は、単に断片的な情報を集めることではなく、**セールス・ワークフローで即座に活用できるデータ**を作成することです。

## 2. チーム構成
チームは3つのサブエージェントで構成されています：
1. `Agent_1_Lead_Sourcing_Collector` (リード探索・収集)
2. `Agent_2_Social_Public_Intelligence` (SNS・公開情報インテリジェンス)
3. `Agent_3_Company_Website_Intelligence` (企業ウェブサイト・インテリジェンス)

各サブエージェントは以下の構造を持ちます：
- `Knowledge/` (ナレッジ)
- `Workflows/` (ワークフロー)
- `Skills/` (スキル)
- `Rules/` (ルール)

## 3. ワークスペース構造

### ルートレベル
- `Agent_1_Lead_Sourcing_Collector/`
- `Agent_2_Social_Public_Intelligence/`
- `Agent_3_Company_Website_Intelligence/`
- `Output/`
- `Global_Guideline.md`
- `README.md`
- `Master_Index.md`

### アウトプット
各リサーチラン（調査実行）には、`Output/` 内に専用のディレクトリを作成する必要があります。

推奨される構成：
- `Master_Index.md`
- `ICP_Input_Brief.md` (独自のICPブリーフがある場合)
- `00_Input_List.md` (ユーザーが入力リストを提供した場合)
- `01_Lead_List.md`
- `02_Public_Customer_Intelligence.md`
- `03_Company_Website_Intelligence.md`
- `04_Enriched_Record_Summary.md`

リードリストに紐付いたモードの他に、以下の独立した実行もサポートしています：
- Agent 2 による独立したSNS調査
- Agent 3 による独立したウェブサイト調査

意味：
- 数字のプレフィックス（接頭辞）により、閲覧順序が即座に分かります。
- `Master_Index.md` はメインのエントリポイントであり、数字のプレフィックスは必須ではありません。

## 4. 運営原則
- `SaleTeam` 内のすべての内部リンクは**相対パス**を使用すること。
- Markdown出力、ガイドライン、インデックス、レポート内で `d:/Researcher/...` のような絶対パスをハードコードしないこと。
- パスは現在のファイルの場所から開始すること。例：
  - 同一ディレクトリ内のリンクは `./...`
  - 同階層または親ディレクトリへのリンクは `../...`
- **実行前のインテーク (Intake before execution)**: すべての調査実行は、エージェントがデータを処理する前に、必ずインテークステップを通らなければなりません。
- **出典優先**: すべての重要なデータには出典（ソース）を明記すること。
- **公開データの使用**: 原則として公開されているデータのみを使用します。プライベートなデータ、ログインが必要なデータ、または利用規約に違反するデータは収集しません。
- **重複排除の徹底**: 理由なくリードリストに重複したレコードを残してはいけません。
- **実行可能性の追求**: アウトプットは、SDR、AE、ファウンダー主導のセールスモデル、またはカスタマーサクセス (CS) プロセスですぐに使えるものである必要があります。
- **鮮度の重視**: 情報の取得日と時間的感度を明記すること。
- **事実の捏造禁止**: 確認できない場合は、`未確認` とマークすること。
- ユーザー提供のリストから開始する場合、すべてのアウトプットを通じて一貫した `レコードID` を使用すること。
- Agent 2 または Agent 3 が独立して実行する場合、その実行範囲内で一貫したローカルIDの使用が許可されます。
- 信頼できる実行を行うのに情報が不十分な場合は、ユーザーに再確認すること。
- 追加の処理や補助的なアウトプットは、ユーザーからの要求がある場合にのみ提案すること。
- ユーザーが日本語で作業している場合、`SaleTeam` のアウトプットは原則として日本語で記述すること。

## 4A. インテーク（受付）規則
- インテークでは、少なくとも以下の事項を明確にする必要があります：
  - 調査の目的
  - 現在ある入力データ
  - 入力ソースの種類
  - 希望するアウトプットレベル
- ICP（理想的な顧客像）や優先基準がある場合は、最初から記録しておくこと。
- ICPが未定の場合は `未特定` と記録できますが、勝手にICPを捏造してはいけません。
- 重要なデータが不足している場合、Agent 1 はリード作成や拡充に進む前に再確認を行う必要があります。
- 「企業リスト」が入力された場合、デフォルトで `アカウントレベルの入力` と見なします。
- 「入札情報」、「資金調達」、「企業のマーケティング活動」、「求人情報」などは、証拠が伴う企業リストとして提示できる場合、Agent 1 の有効な情報源となります。
- `LinkedIn` も、調査範囲（スコープ）が明確に定義されている場合、有効な情報源となります。

## 5. ユーザー提供リストからの拡充フロー
1. インテークを実行し、目的、入力データ、アウトプットレベルを確定。
2. ユーザーが入力リストを提供。
3. Agent 1 がリスト構造を確認し、ICPブリーフを受け取り、標準化して `レコードID` を付与。
4. Agent 1 が `00_Input_List.md` と `01_Lead_List.md` を作成。
5. Agent 1 が各レコードに `拡充ステータス (Enrichment Status)` を付与。
6. Agent 2 による処理：
   - Agent 1 からの適格なレコードに基づいてSNS/公開情報をスキャン（レコードIDを維持）。
   - または、ユーザーからの直接的な指示により独立して実行。
7. Agent 3 による処理：
   - Agent 1 のリストにあるドメインやウェブサイトに基づいて調査（レコードIDを維持）。
   - または、ユーザーからの直接的な指示により独立して実行。
8. Agent 1 または最終統合レイヤーが `04_Enriched_Record_Summary.md` を作成し、情報を統合。

## 6. 各エージェントの役割

### Agent 1: Lead Sourcing Collector
- `SaleTeam` のインテーク責任者。
- 外部ソースからのリードやユーザー提供リストの受付。
- 入札、資金調達、マーケティング、求人などの企業リストソースの処理。
- LinkedIn からのリード獲得（プロファイルのスーシング）。
- データの標準化、レコードIDの付与、重複排除。
- `ICP適合性` の初期判定、拡充ステータスの付与。
- バッチごとの運用メタデータとレコード統合履歴の管理。
- `00_Input_List.md`, `01_Lead_List.md`, `04_Enriched_Record_Summary.md` の作成。

### Agent 2: Social Public Intelligence
- SNS上の公開データの読み取り。
- ユーザーからの直接指示による独立実行のサポート。
- LinkedIn, Facebook, X, Instagram などのプラットフォーム調査。
- 購入シグナル、採用シグナル、インタラクション、関心事の探索。
- 重要プロファイルの継続的な追跡と変更履歴の管理。
- レコードID（Agent 1継承）またはSNSオブジェクトIDの管理。
- シグナルの優先順位付けとフォローアップの管理。
- `02_Public_Customer_Intelligence.md` の作成。

### Agent 3: Company Website Intelligence
- 特定企業のウェブサイトの読み取り。
- ユーザーからの直接指示による独立実行のサポート。
- サイトマップを活用した効率的なスキャン。
- 企業プロファイル、ICPの兆候、製品・価格・メッセージの兆候の抽出。
- ウェブサイト・シグナルの優先順位付けとチェンジログ（変更履歴）の管理。
- レコードID（Agent 1継承）またはウェブサイトオブジェクトIDの管理。
- `03_Company_Website_Intelligence.md` の作成。

## 7. 推奨されるスキャン頻度

### Agent 1
- 定期的なリードソース監視や日次の新規リード受け入れには `Daily_Lead_Source_Scan.md` を使用。
- 週単位での変更確認、クリーニング、ICP適合性の再確認には `Weekly_Lead_Source_Scan.md` を使用。

### Agent 2
- 優先アカウントの監視やアプローチタイミングの把握には `Daily_Social_Signal_Scan.md` を使用。
- シグナルの統合、行動のクラスター化、次週のウォッチリスト更新には `Weekly_Social_Intelligence_Scan.md` を使用。

### Agent 3
- デフォルトでは毎日のスキャンは不要。
- ターゲットアカウントや動きのあるアカウントに対して `Weekly_Website_Change_Scan.md` を使用。
- メッセージ変更、新製品発売、価格改定などの大きなイベント時には、スケジュール外での実行が可能。

## 8. 品質基準
- 識別するための最小限の情報が不足している場合、リードを作成しない。
- 推測を事実として記録しない。
- 証拠を示さずにセールス・インサイトを提供しない。
- 情報が公開情報のみで根拠が弱い場合、信頼レベルを低く設定する。
- 最初の情報源が極端に不足している場合、`入力ソース不足` と明記する。
- インテークが不十分なまま、勝手に拡充ステップに進まない。
