# SaleTeam: 総合ガイドライン

## 1. 目標
`SaleTeam` は、以下の用途のためにデータの収集、標準化、および充実化を専門に行うチームです：
- リード生成
- アカウント（企業）調査
- セールス活動のパーソナライズ
- 顧客の育成（ナーチャリング）とケア

目標は単にバラバラな情報を集めることではなく、セールスワークフローですぐに活用できるデータを作成することです。

## 2. チーム構成
チームは3つのサブエージェントで構成されています：
1. `Agent_1_Lead_Sourcing_Collector`
2. `Agent_2_Social_Public_Intelligence`
3. `Agent_3_Company_Website_Intelligence`

各サブエージェントは以下の構造を持ちます：
- `Knowledge/`（知識）
- `Workflows/`（ワークフロー）
- `Skills/`（スキル）
- `Rules/`（ルール）

## 3. ワークスペース構造

### ルートレベル
- `Agent_1_Lead_Sourcing_Collector/`
- `Agent_2_Social_Public_Intelligence/`
- `Agent_3_Company_Website_Intelligence/`
- `Output/`
- `Global_Guideline.md`
- `README.md`
- `Master_Index.md`

### 出力（Output）
各リサーチ実行（run）ごとに `Output/` 内に独自のディレクトリを作成する必要があります。

推奨される命名規則：
- `Master_Index.md`
- `ICP_Input_Brief.md`（個別のICPブリーフがある場合）
- `00_Input_List.md`（ユーザーが入力リストを提供した場合）
- `01_Lead_List.md`
- `02_Public_Customer_Intelligence.md`
- `03_Company_Website_Intelligence.md`
- `04_Enriched_Record_Summary.md`

リードリストに基づくモード以外に、以下の独立実行もサポートしています：
- Agent 2 による独立したソーシャルリサーチ
- Agent 3 による独立したウェブサイトリサーチ

意味：
- 数字のプレフィックスにより、読み手は読む順番を即座に把握できます。
- `Master_Index.md` は主要なエントリーファイルであり、プレフィックスは必須ではありません。

## 4. 運用原則
- `SaleTeam` 内のすべての内部リンクは相対パスを使用しなければなりません。
- Markdown出力、ガイドライン、インデックス、レポートにおいて、`d:/Researcher/...` のような絶対パスをハードコードしてはいけません。
- パスは現在のファイルの場所から開始する必要があります：
  - 同一ディレクトリ内のリンクには `./...`
  - 同階層または親ディレクトリのリンクには `../...`
- 実行前のインテイク：すべての実行は、対応するエージェントがデータを処理する前にインテイクステップを通過しなければなりません。
- ソース優先：すべての重要なデータにはソース（情報源）が必要です。
- デフォルトで公開データのみを使用：プライベートなデータ、ログインが必要なデータ、利用規約に違反するデータは収集しません。
- 重複排除は必須：理由なくリードリストに重複レコードを残してはいけません。
- アクション性を優先：出力は、SDR、AE、創業者主導の販売モデル、またはCSプロセスで活用できるものでなければなりません。
- 鮮度は非常に重要：情報の取得日と時間的感度を明記しなければなりません。
- 事実を捏造しない：確認できない場合は `未確認` とマークしなければなりません。
- ユーザー提供のリストから開始する場合、出力全体で一貫した `レコードID` を使用しなければなりません。
- Agent 2 または Agent 3 が独立して実行する場合、その実行範囲内で一貫したローカルIDを使用できます。
- 信頼できる実行のために情報が不足している場合は、ユーザーに再確認しなければなりません。
- 追加の処理方法や補助的な出力は、ユーザーの要求がある場合にのみ提案してください。
- ユーザーが日本語で作業している場合、`SaleTeam` のデフォルトの出力は日本語で記述されなければなりません。
- ユーザーの要求や技術的な制約がない限り、出力をASCIIに変換しないでください。

## 4A. インテイクルール
- インテイクでは最低限以下を明確にしなければなりません：
  - 実行目標
  - 現在の入力内容
  - 入力ソースの種類
  - ユーザーが希望する出力レベル
- ICP（理想的な顧客像）や優先基準がある場合は、最初から記録しなければなりません。
- ICPがない場合は `未特定` と記録できますが、勝手にICPを捏造してはいけません。
- 重要なデータが不足している場合、Agent 1 はリード作成や充実化に進む前に再確認しなければなりません。
- 入力が `企業リスト` の場合、デフォルトで `アカウントレベルのインテイク` とみなします。
- `入札`、`資金調達`、`企業のマーケティングシグナル`、`採用` は、ソースの証拠がある企業リストとして表示できる場合、Agent 1 の有効なソースとなります。
- `LinkedIn` も、実行が `LinkedInからのリード収集` と明確に定義され、具体的なスキャン範囲がある場合は有効なソースとなります。

## 5. ユーザー提供リストからの充実化フロー
1. インテイクを実行し、目標、入力、出力レベルを確定する。
2. ユーザーが入力リストを提供する。
3. Agent 1 がリスト構造をチェックし、ICPブリーフ（あれば）を受け取り、標準化して `レコードID` を割り当てる。
4. Agent 1 が `00_Input_List.md` と `01_Lead_List.md` を作成する。
5. Agent 1 が各レコードに `充実化ステータス` を付与する。
6. Agent 2 が以下を行う：
   - Agent 1 からの条件を満たすレコードに基づきソーシャル/公開情報をスキャンし、`レコードID` を維持する。
   - または、ユーザーからの直接のブリーフに基づき独立して実行する。
7. Agent 3 が以下を行う：
   - Agent 1 のリスト内のドメインまたはウェブサイトに基づきウェブサイトをスキャンし、`レコードID` を維持する。
   - または、ユーザーからの直接のブリーフに基づき独立して実行する。
8. Agent 1 または最終統合レイヤーが `04_Enriched_Record_Summary.md` を作成し、レコードごとに情報を統合する。

## 6. 各エージェントの役割

### Agent 1
- `SaleTeam` の入力インテイクを担当
- ユーザー提供のリードソースまたはリストの受け取り
- 入札、資金調達、マーケティング、採用サイトからの `企業リスト` 形式のソースの処理
- ユーザーがプロファイルをソーシングしたい場合の LinkedIn からのリードリスト取得を担当
- データの標準化
- `レコードID` の割り当て
- 重複排除
- `ICP適合度` の暫定評価
- `充実化ステータス` の付与
- バッチごとの運用メタデータとレコード統合履歴の保持
- 必要に応じて `00_Input_List.md`、`01_Lead_List.md`、`04_Enriched_Record_Summary.md` を作成

### Agent 2
- ソーシャルネットワーク上の公開データの読み取り
- ユーザーの直接指定による独立実行を許可
- リードリストとの連携が必要な場合、Agent 1 からの引き継ぎ（handoff）をサポート
- プラットフォームの制限がない場合、デフォルトで `LinkedIn`、`Facebook`、`X`、`Instagram` を調査
- 購買シグナル、採用シグナル、インタラクションシグナル、関心のあるコンテンツの特定
- 重要なプロファイルの経時的な追跡
- プロファイルのベースライン作成と変更ログの保持
- プロファイルレベルでのスキャン頻度管理。アカウントレベルでは各プロファイルの最新情報を採用
- Agent 1 から受け取った場合は `レコードID` を保持し、独立実行の場合は `ソーシャルオブジェクトID` を使用
- 顧客ケアや育成のためのシグナル優先度とアカウント優先度の評価
- ウォッチリストに基づく追跡ステータスとフォローアップの更新
- `02_Public_Customer_Intelligence.md` の作成

### Agent 3
- 特定企業のウェブサイトの読み取り
- ユーザーの直接指定による独立実行を許可
- リードリストとの連携が必要な場合、Agent 1 からの引き継ぎをサポート
- ウェブサイトにサイトマップがある場合は、常にサイトマップから開始
- スキャン前に Agent 1 からの引き継ぎ内容を確認
- 初回スキャン時にウェブサイトのベースラインを作成
- 企業プロファイル、ICPの兆候、製品の兆候、価格の兆候、メッセージングの兆候を抽出
- ウェブサイトシグナルの優先度付与、変更ログの記録、必要に応じたベースラインの更新
- Agent 1 から受け取った場合は `レコードID` を保持し、独立実行の場合は `ウェブサイトオブジェクトID` を使用
- `03_Company_Website_Intelligence.md` の作成

## 7. 推奨スキャン頻度

### Agent 1
- 頻繁に更新されるリードソースを監視している場合や、毎日新しいバッチを受け取る場合は `Daily_Lead_Source_Scan.md` を使用。
- 週単位で変更の確定、クリーニング、ICP適合度の再確認を行う場合は `Weekly_Lead_Source_Scan.md` を使用。

### Agent 2
- 優先ウォッチリストのアカウント（特にケアやアプローチのタイミングを見極める必要がある場合）には `Daily_Social_Signal_Scan.md` を使用。
- シグナルの統合、行動のクラスタリング、翌週のウォッチリスト更新には `Weekly_Social_Intelligence_Scan.md` を使用。

### Agent 3
- デフォルトでは毎日のスキャンは不要。
- ターゲットアカウントや変動の兆候があるアカウントには `Weekly_Website_Change_Scan.md` を使用。
- 各スキャンはサイトマップから開始し、サイトマップがない場合はフォールバック構造を使用。
- メッセージングの変更、新製品の発売、価格変更などの大きなイベントがある場合は、スケジュール外で実行可能。

## 8. 品質ルール
- 特定に必要な最小限の情報が不足している場合はリードを作成しない。
- 推測を事実として記録しない。
- 証拠を示さずにセールスインサイトを提供しない。
- 情報が公開されているだけで弱い場合は、信頼レベルを低く設定する。
- Agent 1 の初期ソースが著しく不足している場合は、`入力ソース不足` と明記する。
- インテイクが不十分な場合、勝手に充実化ステップに移行してはいけない。
